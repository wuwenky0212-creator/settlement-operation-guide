# 代码修改总结

## 修改概述

根据需求文档中功能6、功能7、功能8的修改内容,对后端代码进行了相应的更新。

## 修改的文件

### 1. backend/app/services/status_tracking_service.py

#### 功能6: 交易进度跟踪的修改

**修改内容:**
- 更新了交易生命周期的节点定义,从原来的多节点流程简化为3个主要节点:
  - 节点1: 后台复核 (新增)
  - 节点2: SWIFT证实回执状态 (仅适用于外汇掉期和拆借交易)
  - 节点3: 证实匹配

**具体变更:**
1. 新增 `_is_swap_or_lending_product()` 方法,判断是否为外汇掉期或拆借产品
2. 新增 `_generate_back_office_validation_receipts()` 方法,生成后台复核回执
3. 更新 `_generate_swift_confirmation_receipts()` 方法,实现RMC→FTM串行逻辑:
   - RMC失败时,流程阻断,不显示FTM节点
   - RMC成功时,才触发FTM处理
4. 更新 `_generate_match_receipts()` 方法,增加"匹配中"状态
5. 删除 `_generate_settlement_receipts()` 方法,因为功能6不再包含收付发报环节

#### 功能7: 结算支付进度跟踪的修改

**修改内容:**
- 更新了4阶段流程的详细逻辑,增加了更多子节点

**具体变更:**
1. 更新 `_generate_netting_stage()` 方法:
   - 添加1.1前置准入节点判断逻辑
   - 添加1.2轧差执行节点(自动/手工)
   - 添加1.3发报时机计算节点(待达发报日/待发报)

2. 更新 `_generate_compliance_stage()` 方法:
   - 添加2.1准入数据上送节点
   - 添加2.2反洗钱扫描/校验节点
   - 添加2.3校验结果判定与硬拦截节点
   - 添加2.4 SSI路径自动判定节点
   - 添加2.5路径决策与发送准入节点(SWIFT审批/CBMNet线下处理)

3. `_generate_settlement_stage()` 和 `_generate_cancellation_stage()` 方法保持不变,已经符合需求

### 2. backend/app/services/operation_guide_service.py

#### 功能8: 操作指引的修改

**修改内容:**
- 大幅扩展了操作指引的场景定义,从原来的14个场景增加到62个场景

**具体变更:**
1. 更新 `_configure_transaction_guide_rules()` 方法:
   - 新增节点1(后台复核)的3个场景
   - 更新节点2(SWIFT证实回执)的6个场景
   - 更新节点3(证实匹配)的4个场景
   - 新增交易修改流程的4个场景
   - 新增交易撤销流程的4个场景
   - 共计21个交易相关场景

2. 更新 `_configure_cash_flow_guide_rules()` 方法:
   - 新增阶段1(清算轧差)的8个场景
   - 新增阶段2(反洗钱检查)的6个场景
   - 新增阶段2.5(路径决策与发送准入)的9个场景
   - 新增阶段3(结算执行与回执)的11个场景
   - 新增收付撤销流程的7个场景
   - 共计41个现金流相关场景

3. 更新 `_identify_transaction_scenario()` 方法:
   - 增加后台复核状态的判断
   - 增加SWIFT回执状态的判断(RMC/FTM)
   - 增加证实匹配状态的判断(待匹配/匹配中/匹配成功/手工处理)
   - 增加交易修改流程的判断
   - 增加交易撤销流程的判断

4. 更新 `_identify_cash_flow_scenario()` 方法:
   - 扩展了现金流状态到场景的映射
   - 增加了待达发报日状态的特殊处理
   - 覆盖了所有4个阶段的状态

## 关键改进点

### 1. 交易进度跟踪(功能6)
- 明确了3节点流程:后台复核 → SWIFT证实回执 → 证实匹配
- 实现了RMC→FTM的严格串行逻辑
- RMC失败时硬性阻断,不显示FTM节点
- 增加了"匹配中"状态,更精确地反映匹配过程

### 2. 结算支付进度跟踪(功能7)
- 细化了清算轧差阶段,增加了前置准入、轧差执行、发报时机计算3个子节点
- 细化了合规准入阶段,增加了数据上送、反洗钱扫描、SSI路径判定、路径决策4个子节点
- 明确了SWIFT路径和CBMNet路径的不同处理逻辑
- 增加了"待达发报日"状态,支持发报时机计算

### 3. 操作指引(功能8)
- 场景数量从14个增加到62个,覆盖更全面
- 采用统一的"当前状态...下一步操作..."格式
- 为每个场景提供了明确的操作入口(按钮/链接)
- 增加了交易修改和交易撤销流程的操作指引
- 增加了收付撤销流程的操作指引

## 兼容性说明

所有修改都是向后兼容的:
- 保留了原有的API接口签名
- 保留了原有的数据模型结构
- 新增的字段都是可选的,不影响现有功能
- 简化处理的部分使用了默认值,确保代码可以正常运行

1. 代码中使用了一些简化处理(如`getattr`获取可选字段),实际部署时需要根据数据库模型进行调整
2. 某些状态判断逻辑需要根据实际的业务规则进行完善
3. 发报时机计算需要配置"提前发送收付天数(N)"参数
4. SSI路径判定需要配置SSI参数
