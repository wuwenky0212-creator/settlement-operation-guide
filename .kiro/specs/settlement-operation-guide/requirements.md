# 产品需求文档 (PRD)

## 功能概述

操作指导是GIT-RG系统中的一个核心功能模块（Settlement RN#17），用于支持按多种条件查询交易确认书并显示最新状态。该功能能够标识交易当前所处的生命周期阶段，如已证实、已清算处理待发报、已发报待接收回执等状态，并提供操作指引帮助用户进行后续处理。

## 术语表

- **交易流水号**: 唯一标识一笔交易的编号
- **发送方参考号 (Sender Reference)**: SWIFT报文中的发送方参考号
- **运营机构**: 执行交易的机构
- **方向**: 交易方向（收/付）
- **现金流内部ID**: 系统内部的现金流标识
- **收付信息ID**: 收付款信息的唯一标识
- **结算内部ID**: 结算记录的内部标识
- **RMC**: SWIFT报文路由管理中心
- **FTM**: SWIFT报文传输消息
- **SWIFT证实回执**: SWIFT系统返回的确认回执
- **证实匹配状态**: 交易确认的匹配结果状态
- **反洗钱**: 反洗钱系统检查
- **核心**: 核心银行系统
- **内部账**: 内部账户系统
- **HKSWIFT**: 香港SWIFT系统
- **pacs.008**: ISO 20022 客户贷记转账报文
- **pacs.009**: ISO 20022 金融机构贷记转账报文
- **camt.057**: ISO 20022 通知报文
- **camt.056**: ISO 20022 取消请求报文
- **camt.058**: ISO 20022 取消响应报文

## 功能需求

### 功能1: 交易汇总列表

#### 1. 功能点描述
提供交易汇总列表查询功能，支持多种查询条件组合，以列表形式展示交易记录，支持分页浏览，用户可以快速定位和查看交易。

#### 2. 前提与约束
- 用户必须拥有交易查询权限
- 支持多条件组合查询
- 查询结果支持分页显示
- 默认按交易日降序排列

#### 3. 输入描述
查询条件：
- **外部流水号** (必填): 字符串
- **交易状态** (可选): 下拉选择，枚举值包括：
  - 生效
  - 到期
  - 失效
- **交易日** (可选): 日期或日期范围，格式 YYYY-MM-DD
- **起息日** (可选): 日期或日期范围，格式 YYYY-MM-DD
- **到期日** (可选): 日期或日期范围，格式 YYYY-MM-DD
- **交易对手** (可选): 字符串，交易对手方名称或代码
- **产品** (可选): 下拉选择，枚举值包括：
  - 外汇即期
  - 外汇远期
  - 外汇掉期
  - 同业拆借
  - 货币市场存款
  - 现券买卖
  - 买断式回购
  - 质押式回购
  - 单边现金流
- **货币** (可选): 字符串，3位ISO货币代码（如 CNY、USD、EUR、HKD 等）
- **运营机构** (可选): 下拉选择，枚举值包括：
  - 1530H_中国银行(香港)有限公司
  - 052_中国银行（香港）有限公司文莱分行
  - 053_中国银行（香港）有限公司仰光分行
- **清算方式** (可选): 下拉选择，枚举值包括：
  - 全额
  - 净额
  - 集中
  - 无需
  - 我行代理
  - 他行代理
- **证实方式** (可选): 下拉选择，枚举值包括：
  - SWIFT
  - 文本
  - 无证实
- **交易来源** (可选): 下拉选择，枚举值包括：
  - GIT
  - FXO
  - FXS
  - FXY
  - FXW

#### 4. 输出描述
成功时返回：
- **交易列表**: 当前页的交易记录，每条记录包括：
  - **外部流水号**: 外部系统的交易流水号
  - **交易流水号**: 系统内部交易流水号
  - **录入日**: 交易录入系统的日期
  - **交易日**: 交易发生日期
  - **起息日**: 起息日期
  - **到期日**: 到期日期
  - **账户**: 交易账户信息
  - **产品**: 产品类型（如外汇即期、外汇远期等）
  - **买卖方向**: 买入或卖出
  - **标的物**: 交易标的物描述
  - **交易对手**: 交易对手方名称
  - **交易状态**: 当前交易状态
  - **后线处理状态**: 后线处理的当前状态
  - **清算方式**: 清算方式（如全额、净额、集中等）
  - **证实编号**: 证实确认的编号
  - **证实方式**: 证实方式（如SWIFT、文本、无证实）
  - **证实匹配方式**: 证实匹配的方式
  - **交易性质**: 交易性质描述
  - **交易来源**: 交易来源系统（如GIT、FXO等）
  - **事件类型**: 最新事件类型
  - **运营机构**: 运营机构名称
  - **交易员**: 交易员姓名
  - **操作**: 操作按钮
- **分页信息**: 
  - **当前页码**: 当前页数
  - **总页数**: 总页数
  - **记录总数**: 符合条件的记录总数
  - **当前页记录数**: 当前页显示的记录数
- **查询条件摘要**: 使用的查询条件列表

无匹配记录时返回：
- **空列表**: []
- **记录总数**: 0
- **提示信息**: "未找到符合条件的交易记录"

参数错误时返回：
- **错误信息**: 具体的参数错误描述
- **无效参数列表**: 列出所有无效的参数及原因

#### 5. 验收标准
1. 系统支持通过任意查询条件组合进行交易查询
2. 查询结果以列表形式展示，默认按交易日期降序排列
3. 支持分页显示，用户可以选择每页显示15/20/50/100条记录
4. 分页控件包含首页、上一页、下一页、末页、页码跳转功能
5. 列表显示当前页记录数和总记录数（如"显示1到20，共82记录"）
6. 用户可以点击列表中的任意交易记录查看详情
7. 支持导出查询结果为Excel或CSV格式

### 功能2: 交易详情查看

#### 1. 功能点描述
提供交易详细信息查看功能，通过多标签页方式展示交易的完整信息，包括交易信息、事件信息、账务信息、支付信息。

#### 2. 前提与约束
- 用户必须具有交易详情查看权限
- 从交易汇总列表点击进入详情页
- 详情页采用标签页（Tab）方式组织信息
- 不同标签页展示不同维度的交易信息

#### 3. 输入描述
- **外部流水号** (必填): 字符串，从交易汇总列表选择的交易标识

#### 4. 输出描述
成功时返回交易详情，包含以下标签页：

**标签页1: 交易信息**
- **交易信息**:
  - 货币对
  - 金额(USD/CNY)：BUY金额、SELL金额（带方向标识和颜色）
  - 即期汇率/点差/成本汇率
  - 分润金额、分润币种
  - 交易日、交易时间
  - 起息日
- **交易账户**:
  - 账户
  - 交易对手（代码和名称）
  - 经纪商
  - 背对背账户
- **备注**:
  - 交易性质
  - 外部流水号
  - 交易目的
  - 备注
- **拆分信息**:
  - 交易拆分：拆分货币对（复选框）
- **拆展字段**:
  - 我方收结算路径(USD)
  - 对手方付结算路径(USD)
  - 我方付结算路径(CNY)
  - 对手方收结算路径(CNY)
  - 清算方式

**标签页2: 事件信息**
- 事件流转记录列表（详见功能3）

**标签页3: 支付信息**
- 支付方式、支付账户信息等（详见功能4）

**标签页4: 账务信息**
- 账务处理详细信息（详见功能5）

未找到记录时返回：
- **错误信息**: "未找到对应的交易记录"
- **交易流水号**: 用户请求的交易流水号

#### 5. 验收标准
1. 从交易汇总列表点击交易记录后，系统打开交易详情页
2. 详情页采用标签页方式组织，包含4个标签页
3. 默认显示"交易信息"标签页
4. 用户可以自由切换标签页查看不同维度的信息
5. 每个标签页的信息完整、准确、实时
6. 详情页提供关闭按钮，关闭后返回交易汇总列表
7. 详情页加载时间在正常负载下不超过300毫秒

### 功能3: 事件信息查看

#### 1. 功能点描述
在交易详情的"事件信息"标签页中，以列表形式展示交易的所有事件流转记录，包括事件类型、发生时间、操作用户等信息，支持事件历史追溯。

#### 2. 前提与约束
- 用户必须具有事件信息查看权限
- 事件记录按时间倒序排列（最新的在前）
- 事件记录不可修改或删除
- 支持分页显示

#### 3. 输入描述
- **外部流水号** (必填): 字符串，从交易详情页自动传入，无需用户手动输入

#### 4. 输出描述
成功时返回：
- **事件记录列表**: 包含所有事件记录，每条记录包括：
  - **外部流水号**: 外部系统的交易流水号
  - **交易流水号**: 系统内部交易流水号
  - **父交易流水号**: 父交易的流水号（如有）
  - **产品**: 产品类型（如外汇即期、外汇远期等）
  - **账户**: 交易账户信息
  - **事件类型**: 事件类型（如BOOKED、PAYMENT-DAILY、MATURED等）
  - **交易状态**: 当前交易状态
  - **录入日**: 交易录入系统的日期
  - **交易日**: 交易发生日期
  - **修改日**: 最后修改日期
  - **后线处理状态**: 后线处理的当前状态
  - **证实状态**: 证实确认状态
  - **证实匹配状态**: 证实匹配的结果状态
  - **操作用户**: 执行操作的用户名
- **分页信息**:
  - **当前页码**: 当前页数
  - **总页数**: 总页数
  - **记录总数**: 事件记录总数

无事件记录时返回：
- **空列表**: []
- **提示信息**: "该交易暂无事件记录"

#### 5. 验收标准
1. 事件信息以列表形式展示，包含完整的14个字段信息
2. 事件记录按修改日降序排列（最新的在前）
3. 支持分页显示，默认每页15条记录
4. 分页控件包含首页、上一页、下一页、末页功能
5. 显示当前页记录数和总记录数（如"显示1到4，共4记录"）
6. 事件记录实时更新，反映最新的交易事件
7. 事件信息加载时间在正常负载下不超过200毫秒

### 功能6: 交易进度跟踪

#### 1. 功能点描述
根据交易当前所处的生命周期阶段，实时跟踪交易从后台复核到证实匹配完成的状态进度，以可视化方式展示交易流程的完成情况。交易进度跟踪严格遵循后台复核 → SWIFT证实回执 → 证实匹配的顺序流转。

**展示位置**:
- 在交易详情页面的"交易信息"标签页底部展示交易生命周期进度跟踪组件
- 进度跟踪组件应该嵌入在交易信息内容区域内，而不是作为独立的标签页

#### 2. 前提与约束
- 用户必须具有进度查看权限
- 交易进度包含三个主要节点：后台复核 → SWIFT证实回执状态 → 证实匹配
- SWIFT证实回执状态仅适用于外汇掉期和拆借交易
- SWIFT证实回执内部遵循严格的串行回执逻辑：RMC → FTM
- RMC失败会阻断整个流程，无FTM回执
- FTM失败会触发待办补发提醒
- 进度组件应该与页面内容自然融合，保持一致的视觉风格

#### 3. 输入描述
- **交易流水号** (必填): 字符串，要查询进度的交易标识

#### 4. 输出描述
成功时返回：

**当前进度信息**:
- **当前节点**: 交易所处的具体节点名称
- **当前状态**: 当前节点的状态（复核中/已通过/已删除/发送成功/发送失败/回执失败/匹配成功/匹配失败）
- **产品类型**: 标识是否适用SWIFT证实流程

**节点1: 后台复核 (Back-office Validation)**

该节点作为流程漏斗，决定交易的去留：

1. **复核中 (In-Review)**:
   - 状态：初始状态
   - 触发条件：前台交易达成并同步至后线
   - 时间戳：交易同步时间
   - 描述：等待结算员核对交易要素与清算路径
   - 操作指引：等待结算员完成复核

2. **复核通过 (Approved)**:
   - 状态：已完成
   - 触发条件：结算员执行"确认通过"指令
   - 时间戳：复核通过时间
   - 描述：交易复核通过，触发后续证实流程
   - 后续逻辑：进入"SWIFT证实回执状态"节点

3. **交易已删除 (Deleted/Cancelled)**:
   - 状态：流程终结
   - 触发条件：结算员执行"退回/删除"指令
   - 时间戳：删除时间
   - 描述：交易已被删除，进入归档库
   - 标记：红色，显示"流程已终结"
   - 后续逻辑：流程不再流转

**节点2: SWIFT 证实回执状态 (SWIFT Confirmation Status)**

此节点为核心逻辑块，仅适用于外汇掉期和拆借交易。该节点内部遵循严格的串行回执逻辑，包含两个子节点：

**A. 内部流转逻辑**:

**子节点1: RMC 发送**
- 状态：处理中/成功/失败
- 触发条件：复核通过后自动触发
- 时间戳：发送时间

- **IF SUCCESS (RMC回执成功)**:
  - 描述：RMC 回执成功，自动触发 FTM 发送
  - 标记：绿色
  - 后续逻辑：自动进入子节点2（FTM处理）

- **IF FAILURE (RMC回执失败)**:
  - 描述：RMC 回执失败，这是阻断点
  - 标记：红色，高亮报错
  - 大节点状态：置为"回执失败"
  - 后续逻辑：系统停止后续所有动作，无 FTM 回执
  - 操作指引：报文出口受阻（RMC 失败），请检查报文规范

**子节点2: FTM 处理**
- 前提条件：RMC 必须为 SUCCESS
- 状态：处理中/成功/失败
- 触发条件：RMC 回执成功后自动触发
- 时间戳：发送时间

- **IF SUCCESS (FTM回执成功)**:
  - 描述：FTM 回执成功
  - 标记：绿色
  - 大节点状态：更新为"发送成功"
  - 后续逻辑：解锁下一流程，进入"证实匹配"节点

- **IF FAILURE (FTM回执失败)**:
  - 描述：FTM 回执失败，报文传输中断
  - 标记：黄色，提醒状态
  - 大节点状态：置为"发送失败"
  - 后续逻辑：触发"待办补发"提醒
  - 操作指引：报文传输中断（FTM 失败），请在待办中执行"补发处理"

**B. 操作指引**:

- **正常指引**:
  - 系统正在处理 SWIFT 链路回执，请稍候

- **异常指引（RMC 失败）**:
  - 报文出口受阻（RMC 失败），请检查报文规范

- **异常指引（FTM 失败）**:
  - 报文传输中断（FTM 失败），请在待办中执行"补发处理"

**节点3: 证实匹配 (Confirmation Matching)**

该节点仅在SWIFT证实回执状态大节点成功后进入：

1. **待匹配 (Pending Match)**:
   - 状态：待处理
   - 触发条件：SWIFT 回执大节点成功后自动进入
   - 时间戳：进入时间
   - 描述：等待系统或人工进行匹配处理

2. **匹配中 (Matching)**:
   - 状态：处理中
   - 触发条件：系统开始自动匹配
   - 时间戳：开始匹配时间
   - 描述：系统正在对对手方报文进行自动打分

3. **匹配成功 (Match Success)**:
   - 状态：已完成
   - 触发条件：自动匹配成功
   - 时间戳：匹配成功时间
   - 描述：证实匹配成功，流程闭环
   - 标记：绿色
   - 后续逻辑：交易进度跟踪完成

4. **手工处理 (Manual Processing)**:
   - 状态：待人工处理
   - 触发条件：自动匹配失败
   - 时间戳：转入手工处理时间
   - 描述：匹配失败，需要结算员执行人工匹配或撤销处理
   - 标记：黄色
   - 操作指引：请在待办任务中进行手工匹配或执行撤销处理

**流程可视化**:
- **进度条**: 以横向进度条方式展示三个主要节点和当前位置
- **节点层级**: SWIFT证实回执状态节点内部展示RMC和FTM两个子节点
- **已完成节点**: 标记为绿色，显示完成时间
- **当前节点**: 标记为蓝色并高亮，显示进行中状态
- **未完成节点**: 标记为灰色
- **阻断节点**: 标记为红色，高亮显示"回执失败"，明确标注无后续回执
- **待处理节点**: 标记为黄色，显示"待办补发"或"手工处理"
- **已终结节点**: 标记为红色，显示"流程已终结"

**节点状态汇总**:
- **后台复核**: 复核中 / 复核通过 / 交易已删除
- **SWIFT证实回执状态**: 处理中 / 发送成功 / 发送失败 / 回执失败
  - 子节点 RMC: 处理中 / 成功 / 失败
  - 子节点 FTM: 处理中 / 成功 / 失败（仅当RMC成功时显示）
- **证实匹配**: 待匹配 / 匹配中 / 匹配成功 / 手工处理

未找到记录时返回：
- **错误信息**: "未找到对应的交易记录"

#### 5. 验收标准
1. 系统能够准确识别交易当前所处的节点和状态
2. 交易进度严格按照预定义的顺序展示（后台复核 → SWIFT证实回执状态 → 证实匹配）
3. SWIFT证实回执状态节点仅对外汇掉期和拆借交易显示
4. SWIFT证实回执状态节点内部严格遵循 RMC → FTM 的串行逻辑
5. RMC 失败时，大节点状态置为"回执失败"，界面高亮报错，明确标注"无后续回执"
6. RMC 失败时，不显示 FTM 子节点
7. FTM 失败时，大节点状态置为"发送失败"，触发"待办补发"提醒
8. 对于每个节点和子节点，清晰显示当前状态和时间戳
9. 自动匹配失败时，状态转为"手工处理"，提供操作指引
10. 交易删除后，流程标记为终结，进入归档库
11. 失败或阻塞的节点明确标注原因和操作指引
12. 提供可视化的进度条，直观展示三个主要节点和当前位置
13. SWIFT证实回执状态节点展开显示RMC和FTM两个子节点的状态
14. 状态信息实时更新，延迟不超过10秒
15. 进度组件嵌入在交易信息标签页底部，与页面风格一致
16. 操作指引根据不同状态动态显示（正常指引/RMC失败指引/FTM失败指引）

#### 6. 交易修改流程

交易修改流程根据原交易所处的不同阶段有不同的处理方式。系统需要支持以下六种交易修改场景：

**场景 1：原交易未审核通过**

**前置条件**: 交易处于"复核中"状态，尚未通过后台复核

**处理流程**:
1. 后台结算人员回退交易
2. 前台人员接收回退通知，修改交易
3. 修改后的交易重新发送至后台
4. 参考新交易流程重新进行处理（后台复核 → SWIFT证实回执 → 证实匹配）

**进度跟踪显示**: 
- 原交易标记为"已回退"
- 修改后的交易作为新交易进入流程，从"复核中"状态开始

**场景 2：原交易已审核通过 - 证实报文未发送**

**前置条件**: 交易已通过后台复核，但证实报文尚未发送

**处理流程**:
1. 后台结算人员回退交易
2. 前台人员修改交易
3. 修改后的交易发送至后台，进行审核
4. 按照交易修改事件出账：冲账（原交易）+ 新交易账务
5. 发送证实报文

**进度跟踪显示**:
- 原交易标记为"已回退"
- 修改后的交易进入"复核中"状态
- 账务处理显示：冲账记录 + 新账务记录

**场景 3：原交易已审核通过 - 证实报文已发送未匹配**

**前置条件**: 证实报文已发送，但尚未完成匹配

**处理流程**:
1. 后台结算人员回退交易
2. 前台人员修改交易
3. 修改后的交易发送至后台，进行审核
4. 按照交易修改事件出账：冲账（原交易）+ 新交易账务
5. 证实报文重发送：
   - **报文要素有变化**: 先发送 cancel 报文，然后重发证实报文
   - **报文要素无变化**: 不生成新的证实报文，无需重发
   - **发送机制**: 默认由用户确认后发送，也可配置为系统自动判断后发送（系统参数控制）

**进度跟踪显示**:
- 原交易标记为"已回退"
- 如需发送 cancel 报文，显示 cancel 报文发送状态
- 修改后的交易进入证实报文发送流程
- 账务处理显示：冲账记录 + 新账务记录

**场景 4：原交易已审核通过 - 证实报文已发送已匹配**

**前置条件**: 证实报文已发送且已完成匹配

**处理流程**:
1. 后台结算人员撤销交易对应的证实匹配记录
2. 后台结算人员回退交易
3. 前台人员修改交易
4. 修改后的交易发送至后台，进行审核
5. 按照交易修改事件出账：冲账（原交易）+ 新交易账务
6. 后台结算人员判断电文要素是否有变化：
   - **有变化**: 先发送 cancel 报文，然后重发证实报文
   - **无变化**: 不生成新的证实报文
7. 如已重发证实报文，重新进行匹配

**进度跟踪显示**:
- 原交易的证实匹配记录标记为"已撤销"
- 原交易标记为"已回退"
- 如需发送 cancel 报文，显示 cancel 报文发送状态
- 修改后的交易进入完整流程（审核 → 证实 → 匹配）
- 账务处理显示：冲账记录 + 新账务记录

**场景 5：原交易已审核通过 - 无证实报文未交割**

**前置条件**: 交易无需证实报文，且尚未进行交割

**处理流程**:
1. 后台结算人员回退交易
2. 前台人员修改交易
3. 修改后的交易发送至后台，进行审核
4. 按照交易修改事件出账：冲账（原交易）+ 新交易账务

**进度跟踪显示**:
- 原交易标记为"已回退"
- 修改后的交易进入审核流程
- 账务处理显示：冲账记录 + 新账务记录

**场景 6：原交易已审核通过 - 无证实报文已交割**

**前置条件**: 交易无需证实报文，且已完成交割（收付已处理）

**处理流程**:
1. 收付撤销，自动轧差撤销
2. 后台结算人员回退交易
3. 前台人员修改交易
4. 修改后的交易发送至后台，进行审核
5. 按照交易修改事件出账：冲账（原交易）+ 新交易账务

**进度跟踪显示**:
- 原交易的收付记录标记为"已撤销"
- 原交易的轧差记录标记为"已撤销"
- 原交易标记为"已回退"
- 修改后的交易进入完整流程
- 账务处理显示：冲账记录 + 新账务记录

**交易修改的通用规则**:
1. **回退权限**: 只有后台结算人员有权限回退交易
2. **修改权限**: 前台人员在交易被回退后才能修改
3. **账务处理**: 所有修改场景都需要进行冲账 + 新账务的处理
4. **报文处理**: 
   - 系统自动判断是否需要发送 cancel 报文（基于电文要素是否变化）
   - 支持用户确认或系统自动发送两种模式（参数控制）
5. **进度跟踪**: 
   - 原交易和修改后的交易分别跟踪
   - 原交易标记为"已回退"状态
   - 修改后的交易作为新交易进入相应流程
6. **关联关系**: 系统保留原交易与修改后交易的关联关系，便于追溯

#### 7. 交易撤销流程

交易撤销流程根据原交易所处的不同阶段有不同的处理方式。系统需要支持以下六种交易撤销场景：

**场景 1：原交易未审核通过**

**前置条件**: 交易处于"复核中"状态，尚未通过后台复核

**处理流程**:
1. 后台结算人员回退交易
2. 前台人员撤销交易

**进度跟踪显示**: 
- 原交易标记为"已撤销"
- 流程终止，不产生新交易

- 撤销事件进入"审核中"状态
- 账务处理显示：冲账记录
- 显示 cancel 证实报文发送状态
- 流程终止

**场景 4：原交易已审核通过 - 证实报文已发送已匹配**

**前置条件**: 证实报文已发送且已完成匹配

**处理流程**:
1. 后台结算人员撤销交易对应的证实匹配记录
2. 后台结算人员回退交易
3. 前台人员撤销交易
4. 撤销事件发送至后台，进行审核
5. 后台审核撤销事件
6. 按照交易撤销事件出账：冲账（原交易账务）
7. 后台结算人员发送撤销 cancel 证实报文

**进度跟踪显示**:
- 原交易的证实匹配记录标记为"已撤销"
- 原交易标记为"已撤销"
- 撤销事件进入"审核中"状态
- 账务处理显示：冲账记录
- 显示 cancel 证实报文发送状态
- 流程终止

**场景 5：原交易已审核通过 - 无证实报文未交割**

**前置条件**: 交易无需证实报文，且尚未进行交割

**处理流程**:
1. 后台结算人员回退交易
2. 前台人员撤销交易
3. 撤销事件发送至后台，进行审核
4. 后台审核撤销事件
5. 按照交易撤销事件出账：冲账（原交易账务）

**进度跟踪显示**:
- 原交易标记为"已撤销"
- 撤销事件进入"审核中"状态
- 账务处理显示：冲账记录
- 流程终止

**场景 6：原交易已审核通过 - 无证实报文已交割**

**前置条件**: 交易无需证实报文，且已完成交割（收付已处理）

**处理流程**:
1. 收付撤销，自动轧差撤销
2. 后台结算人员回退交易
3. 前台人员撤销交易
4. 撤销事件发送至后台，进行审核
5. 后台审核撤销事件
6. 按照交易撤销事件出账：冲账（原交易账务）

**进度跟踪显示**:
- 原交易的收付记录标记为"已撤销"
- 原交易的轧差记录标记为"已撤销"
- 原交易标记为"已撤销"
- 撤销事件进入"审核中"状态
- 账务处理显示：冲账记录
- 流程终止

**交易撤销的通用规则**:
1. **回退权限**: 只有后台结算人员有权限回退交易
2. **撤销权限**: 前台人员在交易被回退后才能撤销
3. **审核要求**: 所有撤销事件都需要后台审核
4. **账务处理**: 所有撤销场景都需要进行冲账处理（仅冲账，不产生新账务）
5. **报文处理**: 
   - 如证实报文已发送，需要发送 cancel 证实报文
   - 如证实匹配已完成，需先撤销匹配记录
6. **收付处理**: 
   - 如收付已处理，需要先进行收付撤销和轧差撤销
7. **进度跟踪**: 
   - 原交易标记为"已撤销"状态
   - 撤销事件单独跟踪审核状态
   - 流程终止，不产生新交易
8. **最终状态**: 交易撤销完成后，交易进入归档库，标记为"已撤销"

**交易修改 vs 交易撤销的区别**:
- **交易修改**: 冲账 + 新账务，产生修改后的新交易，流程继续
- **交易撤销**: 仅冲账，不产生新交易，流程终止

### 功能7: 结算支付进度跟踪

#### 1. 功能点描述
根据现金流当前所处的结算支付阶段，实时跟踪从清算轧差到账务入账的完整流程，以可视化方式展示结算执行的完成情况。该功能涵盖清算轧差、合规准入、结算执行与回执、异常补偿四个核心阶段。

**展示位置**:
- 在交易详情页面的"现金流信息"标签页底部展示结算支付进度跟踪组件
- 进度跟踪组件应该嵌入在现金流信息内容区域内，而不是作为独立的标签页

#### 2. 前提与约束
- 用户必须具有进度查看权限
- 流程严格按照预定义的四个阶段顺序执行
- 不同产品类型（外汇/拆借 vs 现券/回购）有不同的前置触发条件
- 不同发送路径（SWIFT vs CBMNet）有不同的处理逻辑
- 某些阶段失败会硬性阻断后续流程
- 进度组件应该与页面内容自然融合，保持一致的视觉风格

#### 3. 输入描述
- **现金流内部ID** (必填): 字符串，要查询进度的现金流标识

#### 4. 输出描述
成功时返回：

**当前进度信息**:
- **当前阶段**: 现金流所处的具体阶段名称（清算轧差/合规准入/结算执行/异常补偿）
- **当前状态**: 当前阶段的状态（待处理/处理中/成功/失败/待审批/已阻断）
- **发送路径**: 标识使用的结算通道（SWIFT/CBMNet/内部账）

**阶段1: 清算轧差 (Netting & Settlement Instruction)**

**目标**: 确定结算头寸与收付指令

**1.1 前置准入节点判断逻辑**:

**判断规则**:
- **外汇/拆借**: 证实匹配成功？(Yes/No)
- **现券/回购**: 文本证实通过？(Yes/No)

**UI 表现**:
- **若未满足**: 节点显示为"等待证实"，流程停滞
- **若已满足**: 节点显示为"证实完成"，流程继续

**状态流转**:
- 未满足前置条件：状态显示为"等待证实"
- 满足前置条件：自动进入轧差执行节点

**1.2 轧差执行节点**:

**节点 A: 自动轧差（系统日终跑批触发）**:
- 触发条件：日终批处理时自动执行
- 处理逻辑：系统按交易对手、币种、结算日自动合并
- 输出物：生成待处理的收付发报记录
- 状态：自动轧差完成

**节点 B: 手工轧差/重组（结算员手工操作触发）**:
- 触发条件：结算员在UI界面手动勾选多笔成交记录
- 处理逻辑：执行"手工轧差"或"重组"操作
- 输出物：生成待处理的收付发报记录
- 状态：手工轧差完成

**1.3 发报时机计算节点**:

**规则逻辑**:
- 系统获取参数：提前发送收付天数 (N)
- 计算公式：发报预定日 = 结算日 - N

**状态流转**:
- **未达发报日**: 
  - 状态显示为"待达发报日"
  - 此时指令已生成，但在"静默期"
  - 流程暂停，等待发报日到达
- **到达发报日**: 
  - 状态自动激活为"待发报"
  - 流程正式向阶段 2（合规准入）推送

**UI 交互建议**:
- 在进度条下方标注一行小字："预计发报日期：YYYY-MM-DD"
- 显示倒计时天数："距离发报日还有 X 天"

**输出状态**: 待达发报日 / 待发报 (Pending Dispatch)

**阶段2: 反洗钱检查 (AML Check)**

**目标**: 在指令发送前进行风险拦截

**2.1 准入数据上送 (Data Submission)**:

**节点定义**: 将结算指令中的交易对手、起息日、金额、路径等关键信息实时推送至反洗钱（AML）系统

**逻辑规则**:
- 触发时机：阶段 1 判定为"待发报"后立即触发
- 数据内容：交易对手信息、起息日、金额、结算路径等关键字段
- 推送方式：实时接口调用

**UI 状态表现**:
- **Success**: 数据发送成功，自动进入下一子节点
- **Processing**: 显示"正在对接反洗钱系统..."
- **Error**: 数据发送失败，显示错误信息

**2.2 反洗钱扫描/校验 (AML Scanning)**:

**节点定义**: 反洗钱系统进行检查

**逻辑规则**:
- 处理方式：反洗钱系统自动执行校验

**UI 状态表现**:
- **Processing**: 进度条处于加载状态，提示"反洗钱扫描中"
- 显示预计处理时间（如有）

**2.3 校验结果判定与硬拦截 (Result & Interception)**:

**节点定义**: 决定流程是"放行"还是"永久阻断"

**逻辑规则**:
- **通过 (Pass)**:
  - 条件：所有校验项均通过
  - 后续动作：自动激活下一阶段（结算发报人工审批）
  - 状态：合规校验通过
  
- **拦截 (Intercept/Fail)**:
  - 条件：任一校验项命中风险规则
  - 后续动作：流程彻底终止，不可继续
  - 状态：反洗钱拦截
  - 拦截信息：
    - 错误码：系统生成的拦截错误码
    - 拦截原因：具体的风险提示（如"交易对手在黑名单中"、"涉及制裁国家"等）
    - 建议操作：联系合规部门进行人工审核

**UI 状态表现**:
- **Success（绿色标识）**:
  - 节点显示"合规校验通过"
  - 流程自动流向阶段 3
  
- **Error（红色高亮）**:
  - 节点显示"反洗钱拦截"
  - 弹出显著的红色警示信息："触发反洗钱硬拦截"
  - 下方显示具体的错误详情：
    - 错误码：[系统错误码]
    - 拦截原因：[具体原因描述]
    - 建议操作：请联系合规部门进行人工审核

**UI 流程图交互建议**:
- **异常可见性**: 当发生拦截时，进度组件不仅要在该节点显示红色，还应在下方提供具体的错误详情（如：错误码、拦截原因），方便结算员第一时间联系合规部门
- **状态描述文案**:
  - 通过时：显示"合规校验通过"
  - 拦截时：显示"已拦截 - 风险待核实"

**阶段2.5: 路径决策与发送准入 (Routing & Approval)**

**目标**: 根据 SSI 参数决定结算通道，并完成相应的审批或确认流程

**2.4 SSI 路径自动判定 (SSI Path Decision)**:

**节点定义**: 根据预设的 SSI（标准结算指令）参数，决定结算通道

**逻辑规则**:
- 输入：SSI 参数（标准结算指令配置）
- 判定逻辑：系统根据 SSI 参数自动判断发送路径
- 判定结果：
  - **SWIFT 路径**: 进入系统自动化发报流程（需经过人工审批）
  - **人工路径（CBMNet）**: 进入线下人工处理流程

**UI 状态表现**:
- **Success**: 节点下方显示确定的路径标签
  - 示例：路径：SWIFT
  - 示例：路径：人工（CBMNet）
- 路径标签使用不同颜色区分（SWIFT：蓝色，人工：橙色）

**2.5 路径决策与发送准入 (Routing & Approval)**:

根据路径判定结果，流程分为两个并行分支：

**分支 A：节点 2.5a - SWIFT 路径人工审批 (SWIFT Sending Approval)**

**节点定义**: 针对 SWIFT 报文的风险控制，触发人工审核

**逻辑规则**:
- 触发条件：路径判定为 SWIFT
- 审批流：进入"收付审批流程"，等待审批人员审批
- 结果判定：
  - **审批通过**: 激活阶段 3 的 RMC/FTM 发送逻辑
  - **审批拒绝**: 流程终止，不可继续

**UI 状态表现**:
- **Pending（黄色标识）**: 
  - 显示"待审批"
  - 标注审批人信息
  - 显示提交审批时间
- **Success（绿色标识）**: 
  - 显示"审批通过"
  - 显示审批人和审批时间
  - 自动向阶段 3 的发报节点流转
- **Error（红色标识）**: 
  - 显示"审批拒绝"
  - 显示拒绝原因
  - 流程阻断，标注"流程已终止"

**分支 B：节点 2.5b - 人工（CBMNet）路径指令导出与线下确认 (CBMNet Offline Processing)**

**节点定义**: 处理线下支付逻辑

**逻辑规则**:
- 触发条件：路径判定为人工处理（CBMNet）
- 动作：
  - 系统生成收付指令
  - 允许结算员"导出收付指令"
  - 结算员在 CBMNet 系统完成线下操作
- 确认逻辑：
  - 结算员在 CBMNet 完成操作后，返回系统点击"人工确认"
  - 人工确认通过后，直接流转至阶段 3 的"核心入账"环节（跳过 RMC/FTM 发送）

**UI 状态表现**:
- **Pending（橙色标识）**: 
  - 显示"待线下处理"
  - 提供"导出收付指令"按钮
  - 提供"人工确认"按钮（待结算员完成线下操作后点击）
- **Success（绿色标识）**: 
  - 显示"人工确认通过"
  - 显示确认人和确认时间
  - 节点变绿，直接流转至阶段 3 的"核心入账"环节
- **Error（红色标识）**:
  - 显示"确认拒绝"或"线下处理失败"
  - 流程终止

**路径分支说明**:
- SWIFT 路径：2.5a（人工审批）→ 阶段 3（RMC/FTM 发送）→ 核心入账
- CBMNet 路径：2.5b（线下处理 + 人工确认）→ 阶段 3（直接核心入账，跳过 RMC/FTM）

**阶段3: 结算执行与回执确认 (Execution & Acknowledgement)**

**前置条件**: 阶段 2.5 路径决策与发送准入已完成（SWIFT 路径审批通过或 CBMNet 路径人工确认通过）

**路径A: 发送路径 = SWIFT**

这是最核心的严谨性环节，必须遵循线性顺序，一旦失败则流程中断。

**A. 传输层回执 (Transmission Layer)**

必须遵循线性顺序，一旦失败则流程中断：

**节点 3.1：传输层 - RMC 发送与回执 (RMC Transmission)**

**节点定义**: SWIFT 报文送达网关的第一步出口校验

**逻辑规则**:
- 触发前提：阶段 2.5 的"SWIFT 路径人工审批"通过
- 发送动作：系统自动发送 RMC 报文至 SWIFT 网关
- 回执判定：
  - **收到成功回执**: 自动激活下一节点（FTM）
  - **回执失败**: 流程阻断，停止后续所有自动化操作

**UI 状态表现**:
- **Processing（蓝色标识）**: 显示"RMC 发送中"，等待回执
- **Success（绿色标识）**: 
  - 显示"已收到 RMC 回执"
  - 自动流向节点 3.2（FTM）
- **Error（红色高亮）**: 
  - 显示"未收到 RMC 回执"
  - 流程阻断，停止后续所有自动化操作
  - 操作指引：提示"报文出口受阻，请检查报文规范或进入撤销环节"

**节点 3.2：传输层 - FTM 发送与回执 (FTM Transmission)**

**节点定义**: 报文正式送至 SWIFT 网络

**逻辑规则**:
- 触发前提：节点 3.1（RMC）成功收到回执
- 发送动作：系统自动发送 FTM 报文至 SWIFT 网络
- 回执判定：
  - **收到成功回执**: 报文正式送出，激活"核心入账"节点
  - **回执失败**: 标记为"失败"，需人工介入

**UI 状态表现**:
- **Processing（蓝色标识）**: 显示"FTM 发送中"，等待回执
- **Success（绿色标识）**: 
  - 显示"已收到 FTM 回执"
  - 报文正式出库，自动流向"核心入账"节点
- **Error（红色标识）**: 
  - 显示"FTM 发送失败"
  - 记录失败日志，需人工介入排查
  - 操作指引：提示"报文传输中断，请在待办中执行'补发处理'或进入撤销环节"

**B. 账务层回执 (Accounting Layer)**

报文出库后，需同步完成核心扣划账处理：

**节点 3.3：账务层 - 发送核心入账 (Core Accounting)**

**节点定义**: 同步触发核心账务系统的资金扣划

**逻辑规则**:
- 触发前提：
  - SWIFT 路径：节点 3.2（FTM）成功收到回执
  - CBMNet 路径：阶段 2.5b 人工确认通过
- 执行动作：
  - 自动模式：系统自动将账务信息推送至核心系统
  - 手动模式：由结算员手动触发推送
- 回执等待：系统等待核心系统返回入账结果

**输出分支（四个子状态）**:

1. **入账处理中 (Processing)**:
   - 状态描述：指令已送达核心系统，等待回执反馈
   - 触发条件：账务信息已成功推送至核心系统
   - 后续动作：等待核心系统处理并返回结果

2. **入账成功 (Settled)**:
   - 状态描述：终态，结算单标记为"结算完成"
   - 触发条件：核心系统返回入账成功回执
   - 后续动作：流程完成，无需后续操作
   - 结算单状态：更新为"结算完成 (Settled)"

3. **入账失败 (Failed)**:
   - 状态描述：回执反馈入账失败
   - 触发条件：核心系统返回入账失败回执
   - 失败原因：显示核心系统返回的具体失败原因
   - 后续动作：需要人工介入处理

4. **入账不明 (Unknown)**:
   - 状态描述：核心系统超时或未给出明确回执
   - 触发条件：
     - 核心系统响应超时
     - 核心系统返回不明确的回执
     - 网络中断导致回执丢失
   - 后续动作：需要人工确认实际入账状态

**UI 状态表现与异常处理**:

- **Processing（蓝色标识）**: 
  - 显示"核心入账处理中"
  - 显示处理进度或等待时间（如有）
  - 提示"等待核心系统处理"

- **Success（绿色标识）**: 
  - 显示"结算完成 (Settled)"
  - 显示入账时间和金额
  - 节点变绿，流程条结束
  - 流程完成，无需后续操作

- **Failed（红色标识）**: 
  - 显示"入账失败"
  - 显示失败原因（核心系统返回的具体错误信息）
  - 节点显示红色
  - **关键交互**：UI 必须强制提供一个【跳转至内部账划转查询】按钮，引导结算员在该功能模块进行人工确认
  - 操作指引：请查询内部账划转情况

- **Unknown（黄色标识）**: 
  - 显示"入账不明"
  - 显示不明原因（如"核心系统超时"、"回执未返回"等）
  - 节点显示黄色
  - **关键交互**：UI 必须强制提供一个【跳转至内部账划转查询】按钮，引导结算员在该功能模块进行人工确认
  - 操作指引：请查询内部账划转情况

**关键交互要求**:
- 当状态为"入账失败"或"入账不明"时，UI 界面必须：
  1. 在节点下方显著位置提供【跳转至内部账划转查询】按钮
  2. 按钮点击后，系统自动跳转至【内部账划转查询】功能模块
  3. 自动带入当前交易的查询条件（如交易流水号、现金流ID等）
  4. 方便结算员快速查询实际入账情况并进行后续处理

**路径B: 发送路径 = CBMNet（人工处理）**

CBMNet 路径跳过 RMC/FTM 发送环节（节点 3.1 和 3.2），直接进入核心扣划账处理（节点 3.3）：

1. **前置条件**:
   - 阶段 2.5b 中结算员已完成线下操作并点击"人工确认"
   - 人工确认通过后，流程直接进入节点 3.3（核心入账处理）

2. **核心扣划账**:
   - 执行逻辑：同节点 3.3 的处理逻辑
   - 状态：核心入账处理中/结算完成/入账失败/入账不明

**路径说明**:
- **SWIFT 路径**: 必须完整经历节点 3.1（RMC）→ 节点 3.2（FTM）→ 节点 3.3（核心入账）的全流程
- **CBMNet 路径**: 跳过节点 3.1 和 3.2，直接进入节点 3.3（核心入账）

#### 4. 收付撤销流程

收付撤销流程适用于起息日或到期日当天及前几日的收付报文撤销处理。系统需要支持收付报文已发送后的撤销处理和撤销后的重做流程。

**适用时间范围**: 
- 起息日当天及起息日前几日
- 到期日当天及到期日前几日

**场景 1：收付报文已发送的撤销处理**

**前置条件**: 收付报文已通过阶段 3 发送至代理行

**处理流程**:
1. 结算人员执行收付撤销处理
2. 系统生成撤销报文（cancel 报文）
3. 结算人员确认撤销报文内容正确
4. **可选审批流程**（如配置为需要审批）:
   - 系统触发撤销审批流程
   - 等待审批人员审批
   - 审批通过后，发送撤销报文给代理行
   - 审批拒绝，撤销操作终止
5. 发送撤销报文至代理行
6. 系统自动更新状态：
   - 收付状态更新为"待收付"
   - 自动轧差回退
   - 轧差状态更新为"待轧差"

**进度跟踪显示**:
- 收付记录标记为"已撤销"
- 轧差记录标记为"已回退"
- 显示撤销报文发送状态
- 如配置审批流程，显示审批状态（待审批/审批通过/审批拒绝）
- 收付状态回退至"待收付"
- 轧差状态回退至"待轧差"

**UI 状态表现**:
- **撤销处理中**: 蓝色标识，显示"正在处理收付撤销"
- **待审批**: 黄色标识，显示"撤销待审批"，标注审批人信息
- **审批拒绝**: 红色标识，显示"撤销审批拒绝"，撤销操作终止
- **撤销报文发送中**: 蓝色标识，显示"撤销报文发送中"
- **撤销成功**: 绿色标识，显示"收付已撤销"，状态回退至"待收付"

**场景 2：撤销后的重做流程 - 方案 1（现金流修改）**

**前置条件**: 收付撤销成功，收付状态已回退至"待收付"

**处理流程**:
1. 发起一笔现金流修改
2. 系统对应的清算记录重新计算
3. 重新进入结算支付进度跟踪流程：
   - 阶段 1：清算轧差（重新轧差）
   - 阶段 2：反洗钱检查
   - 阶段 2.5：路径决策与发送准入
   - 阶段 3：结算执行与回执确认

**进度跟踪显示**:
- 原收付记录标记为"已撤销"
- 修改后的现金流作为新记录进入流程
- 清算记录重新计算
- 从阶段 1 开始重新跟踪进度

**适用场景**:
- 需要修改现金流金额、币种、结算路径等要素
- 需要系统自动重新计算清算记录

**场景 3：撤销后的重做流程 - 方案 2（补做单边现金流）**

**前置条件**: 收付撤销成功，收付状态已回退至"待收付"

**处理流程**:
1. 补做一笔单边现金流
2. 手工清算轧差处理：
   - 结算人员手动勾选需要轧差的现金流记录
   - 执行手工轧差操作
   - 生成收付发报记录
3. 进入结算支付进度跟踪流程：
   - 阶段 1：清算轧差（手工轧差已完成）
   - 阶段 2：反洗钱检查
   - 阶段 2.5：路径决策与发送准入
   - 阶段 3：结算执行与回执确认

**进度跟踪显示**:
- 原收付记录标记为"已撤销"
- 新增的单边现金流作为独立记录
- 手工轧差记录单独显示
- 从阶段 1（手工轧差）开始跟踪进度

**适用场景**:
- 需要补做一笔独立的单边现金流
- 需要手工控制轧差逻辑
- 特殊情况下的收付处理

**收付撤销的通用规则**:
1. **撤销权限**: 只有结算人员有权限执行收付撤销
2. **时间限制**: 仅适用于起息日/到期日当天及前几日
3. **审批配置**: 系统支持配置是否需要审批流程
4. **自动回退**: 撤销成功后，系统自动回退收付状态和轧差状态
5. **报文发送**: 需要发送撤销报文（cancel 报文）至代理行
6. **重做方案**: 支持两种重做方案（现金流修改或补做单边现金流）
7. **进度跟踪**: 
   - 原收付记录标记为"已撤销"
   - 重做后的记录重新进入完整流程
8. **关联关系**: 系统保留原收付记录与重做记录的关联关系

**收付撤销 vs 交易撤销的区别**:
- **交易撤销**: 在交易层面进行撤销，影响整个交易生命周期
- **收付撤销**: 在结算支付层面进行撤销，仅影响收付和轧差环节
- **时间范围**: 收付撤销有明确的时间限制（起息日/到期日前后几日）
- **重做机制**: 收付撤销支持两种重做方案，交易撤销不支持重做

未找到记录时返回：
- **错误信息**: "未找到对应的现金流记录"

#### 5. 验收标准
1. 系统能够准确识别现金流当前所处的结算支付阶段（清算轧差/反洗钱检查/路径决策与发送准入/结算执行与回执确认）
2. 流程严格按照预定义的四个阶段顺序展示
3. 清算轧差阶段 - 前置准入节点判断逻辑：
   - 外汇/拆借产品：系统能够正确判断证实匹配是否成功（Yes/No）
   - 现券/回购产品：系统能够正确判断文本证实是否通过（Yes/No）
   - 未满足前置条件时，节点显示为"等待证实"，流程停滞
   - 满足前置条件后，节点显示为"证实完成"，流程自动继续
4. 清算轧差阶段 - 轧差执行节点：
   - 支持节点 A：自动轧差（系统日终跑批触发）
   - 支持节点 B：手工轧差/重组（结算员手工操作触发）
   - 两种轧差方式均能正确生成待处理的收付发报记录
5. 清算轧差阶段 - 发报时机计算节点：
   - 系统能够正确获取参数"提前发送收付天数 (N)"
   - 系统能够正确计算发报预定日：发报预定日 = 结算日 - N
   - 未达发报日时，状态显示为"待达发报日"，指令处于"静默期"
   - 到达发报日时，状态自动激活为"待发报"，流程正式向阶段 2 推送
   - UI 界面在进度条下方标注预计发报日期（格式：YYYY-MM-DD）
   - UI 界面显示距离发报日的倒计时天数
6. 反洗钱检查阶段 - 准入数据上送节点：
   - 系统能够在阶段 1 判定为"待发报"后立即触发数据上送
   - 系统能够正确提取并推送交易对手、起息日、金额、路径等关键信息
   - UI 界面显示"正在对接反洗钱系统..."处理状态
   - 数据发送成功后自动进入下一子节点
   - 数据发送失败时显示明确的错误信息
7. 反洗钱检查阶段 - 反洗钱扫描/校验节点：
   - 系统能够进行黑名单比对、制裁名单扫描及风险模型计算
   - UI 界面显示"反洗钱扫描中"加载状态
   - 进度条处于加载状态，提供视觉反馈
8. 反洗钱检查阶段 - 校验结果判定与硬拦截节点：
   - 校验通过时，节点显示"合规校验通过"（绿色标识），流程自动流向阶段 3
   - 校验拦截时，节点显示"反洗钱拦截"（红色高亮），流程彻底终止
   - 拦截时弹出显著的红色警示信息："触发反洗钱硬拦截"
   - 拦截时在节点下方显示具体的错误详情：错误码、拦截原因、建议操作
   - 拦截状态显示"已拦截 - 风险待核实"
9. 反洗钱检查阶段 - SSI 路径自动判定节点：
   - 系统能够根据 SSI 参数自动判断发送路径
   - 判定结果包括：SWIFT 路径、人工路径（CBMNet）
   - UI 界面在节点下方显示确定的路径标签（如"路径：SWIFT"或"路径：人工（CBMNet）"）
   - 不同路径使用不同颜色标签区分（SWIFT：蓝色，人工：橙色）
10. 反洗钱检查阶段 - 路径决策与发送准入：
   - 系统根据路径判定结果，正确分流至对应的处理分支
   - SWIFT 路径流程：人工审批 → RMC/FTM 发送 → 核心入账
   - CBMNet 路径流程：线下处理 + 人工确认 → 直接核心入账（跳过 RMC/FTM）
11. 反洗钱检查阶段 - SWIFT 路径人工审批节点（2.5a）：
   - 路径判定为 SWIFT 时，自动触发收付审批流程
   - 待审批状态显示黄色标识，标注审批人信息和提交时间
   - 审批通过后显示绿色标识，自动流转至阶段 3 的 RMC/FTM 发送节点
   - 审批拒绝时显示红色标识，流程终止，显示拒绝原因
12. 反洗钱检查阶段 - CBMNet 路径线下处理节点（2.5b）：
   - 路径判定为人工处理（CBMNet）时，显示"待线下处理"（橙色标识）
   - 系统生成收付指令，提供"导出收付指令"按钮
   - 提供"人工确认"按钮，供结算员完成线下操作后点击
   - 人工确认通过后，节点变绿，直接流转至阶段 3 的核心入账环节
   - 确认拒绝或线下处理失败时，显示红色标识，流程终止
13. 结算执行与回执确认阶段 - SWIFT 路径传输层严格遵循节点 3.1（RMC）→ 节点 3.2（FTM）的串行逻辑
14. 节点 3.1（RMC）发送失败时，状态标记为"未收到 RMC 回执"，流程阻断，明确标注"需人工介入排查"
15. 节点 3.1（RMC）发送失败时，不触发节点 3.2（FTM）处理
16. 节点 3.2（FTM）发送失败时，记录失败日志，状态标记为"FTM 发送失败"，提供"补发处理"操作指引
17. 节点 3.3（核心入账）能够区分自动模式和手动模式
18. 节点 3.3 包含四个子状态：入账处理中/入账成功/入账失败/入账不明
19. 入账成功时，结算单最终状态更新为"结算完成 (Settled)"，节点变绿，流程条结束
20. 入账失败时，节点显示红色，UI 必须提供【跳转至内部账划转查询】按钮
21. 入账不明时，节点显示黄色，UI 必须提供【跳转至内部账划转查询】按钮
22. 【跳转至内部账划转查询】按钮点击后，自动跳转至该功能模块并带入当前交易的查询条件
23. CBMNet 路径跳过节点 3.1 和 3.2，直接进入节点 3.3（核心入账）
24. CBMNet 路径的核心扣划账处理逻辑与 SWIFT 路径的节点 3.3 一致
25. 对于每个阶段和子节点，清晰显示当前状态和时间戳
26. 失败或阻断的节点明确标注原因和操作指引
27. 提供可视化的进度条，直观展示主要阶段和当前位置
28. 已完成节点标记为绿色，当前节点标记为蓝色并高亮，未完成节点标记为灰色，失败节点标记为红色，阻断节点标记为红色并高亮，待审批节点标记为黄色，入账不明节点标记为黄色
29. 状态信息实时更新，延迟不超过10秒
30. 进度组件嵌入在现金流信息标签页底部，与页面风格一致
31. 支持打印或导出进度报告
32. 未找到现金流记录时，返回明确的错误信息

### 功能8: 操作指引

#### 1. 功能点描述
基于交易和支付信息当前的生命周期状态，动态生成操作指引，明确告知用户当前状态和下一步应该执行的操作。操作指引统一采用"当前状态...下一步操作..."的格式，确保用户清晰了解流程位置和后续动作。

#### 2. 前提与约束
- 用户必须具有操作指引查看权限
- 操作指引基于当前状态动态生成
- 不同状态对应不同的操作建议
- 某些状态需要提供人工处理入口
- 操作指引需要区分交易进度跟踪和结算支付进度跟踪两个维度

#### 3. 输入描述
- **外部流水号** (必填): 字符串，要查询交易进度操作指引的交易标识
- **现金流内部ID** (可选): 字符串，要查询结算支付进度操作指引的现金流标识

#### 4. 输出描述
成功时返回：

**操作指引信息**:
- **当前状态**: 明确告知用户当前所处的状态
- **下一步操作**: 明确告知用户下一步应该做什么

### 功能9: 数据持久化与并发控制

#### 1. 功能点描述
确保所有交易数据、事件记录和账务信息持久化存储，并通过并发控制机制保证数据一致性和完整性。

#### 2. 前提与约束
- 所有数据操作必须持久化到数据库
- 支持事务处理，保证数据一致性
- 使用乐观锁或悲观锁机制防止并发冲突
- 查询操作不加锁，允许并发读取
- 更新操作需要获取锁

#### 3. 输入描述
无直接用户输入，系统自动处理数据持久化和并发控制

#### 4. 输出描述
正常操作：
- 操作正常完成，无额外输出

持久化失败：
- **错误代码**: 数据库错误代码
- **错误信息**: 详细的错误描述
- **回滚状态**: 确认操作已回滚

并发冲突：
- **错误代码**: CONCURRENT_CONFLICT
- **错误信息**: "记录正在被其他用户修改，请稍后重试"
- **冲突的记录标识**: 交易流水号
- **建议重试时间**: 建议的重试等待时间（秒）

#### 5. 验收标准
1. 所有数据操作立即持久化到数据库
2. 系统重启后能够完整恢复所有数据
3. 数据持久化失败时，系统回滚内存操作并返回错误信息
4. 多个用户同时更新同一条记录时，系统使用锁机制确保数据一致性
5. 检测到并发冲突时，系统返回冲突错误信息并提示用户重试
6. 多个用户同时查询时，系统允许并发读取操作而不加锁
7. 支持数据库事务，确保复杂操作的原子性
8. 系统能够处理至少50个并发用户的同时访问
